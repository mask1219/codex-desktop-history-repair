# PRD：Codex 桌面端历史对话修复项目

## 1. Summary

本项目用于修复 Codex 桌面端在使用第三方 API、切换账号、切换 provider、升级桌面端版本后，历史会话可能丢失或无法继续对话的问题。目标是建立一套独立于上游 provider 和 Codex 内部存储的本地会话系统，让用户无论怎么切换，历史都不丢，且能继续聊。

本 PRD 的修复范围仅覆盖新方案上线后创建的新线程，不包含对历史存量线程的迁移修复承诺。

## 3. Background

### 3.1 Context

当前在 Codex 桌面端接入第三方 `Responses API` 兼容接口时，容易出现以下问题：
- 对话刷新后历史消失
- 切换 provider 后无法续聊
- 切换官方账号后旧线程像新线程一样断开
- 升级 Codex 桌面端后，本地历史无法恢复
- 流式请求中断后，最后一轮整段丢失

### 3.2 Why Now

随着第三方 API、官方账号、兼容网关混用越来越常见，单纯依赖某个上游的会话状态已经不稳定。只要继续使用多 provider 路由，这个问题就会持续影响真实使用体验。

### 3.3 What Changed

`Responses API` 支持通过 `previous_response_id` 串联多轮对话，但不同第三方网关对该机制的兼容程度不一致。同时，Codex 桌面端的内部会话存储不适合作为唯一真相，因此需要把“历史会话”沉淀为本地独立资产。

## 4. Objective

### 4.1 Objective

构建一个独立、本地可恢复的历史对话系统，保证新方案上线后创建的新线程在以下场景中历史仍可见且可继续对话：
- 切换第三方 provider
- 切换官方账号
- 升级 Codex 桌面端
- 流式中断、崩溃、断网

### 4.2 Why It Matters

对用户来说，“历史丢失”会直接破坏信任。哪怕底层数据还在，只要列表空了、标题没了、上下文断了，用户就会认为产品不可靠。

### 4.3 Strategic Value

该项目可以把会话稳定性从上游 provider 的兼容性问题中解耦出来，降低未来接入新 provider、新账号体系时的维护成本。

### 4.4 Key Results

- KR1：以新方案上线后创建的文本线程为统计口径，100% 的线程在切换 provider 后仍可显示。
- KR2：以新方案上线后创建的文本线程为统计口径，100% 的线程在切换账号后仍可显示并继续对话。
- KR3：以新方案上线后创建的文本线程为统计口径，在删除所有远端 `response_id` 后，至少 95% 的线程仍能正常续聊。
- KR4：以新方案上线后创建的文本线程为统计口径，桌面端升级后线程恢复成功率达到 99%。
- KR5：流式中断后，100% 的最后一轮消息保留为 `partial` 或 `failed`，不出现整段消失。
- KR6：对长上下文文本线程，系统支持在本地摘要后继续对话，摘要续聊成功率不低于 90%。

### 4.5 In Scope

- 新方案上线后创建的新线程
- 文本对话线程
- 长上下文文本线程
- 切换 provider、切换账号、升级桌面端、断网/崩溃/流式中断后的本地恢复和续聊

### 4.6 Out of Scope

- 新方案上线前历史线程的迁移、回填、修复
- tool call 线程的跨 provider 兼容续聊
- 图片或其他多模态消息的跨 provider 兼容续聊
- 云端同步、多设备同步

## 5. Market Segment(s)

### 5.1 Primary Users

- 在 Codex 桌面端中混用多个第三方 API 的高级用户
- 需要在多个官方账号之间切换的用户
- 对长对话、代码上下文、连续协作依赖较强的用户

### 5.2 User Constraints

- 用户不关心底层 provider 的会话细节，只关心“历史还在不在”
- 用户希望切换账号和升级版本后仍然延续之前的工作
- 用户希望必要时能导出、备份、恢复自己的历史

## 6. Value Proposition(s)

### 6.1 User Jobs

- 我想在换 provider 后继续原来的对话
- 我想在换账号后还能看到旧线程
- 我想升级 Codex 桌面端后不丢工作记录
- 我想在出错后仍能找回最后一轮内容

### 6.2 Gains

- 历史线程稳定存在
- 线程可持续追问，不被 provider 边界打断
- 对本地数据和备份有掌控感

### 6.3 Pain Relief

- 解决刷新后历史消失
- 解决切换 provider 后上下文断掉
- 解决升级后“像重装了一样”的历史缺失

### 6.4 Differentiation

相比依赖远端会话状态的做法，本方案以本地 transcript 为唯一真相，更适合多 provider 和多账号混用的真实场景。

## 7. Solution

### 7.1 UX / User Flow

用户使用流程：
1. 用户在 Codex 桌面端创建或继续一个线程
2. 系统先把用户消息写入本地存储
3. 系统根据当前 provider / 账号决定是否复用远端链路
4. 如果不能复用，则自动用本地 transcript 重建上下文并发送
5. 如果本地 transcript 超出当前模型上下文限制，则先在本地生成摘要，再用“摘要 + 最近消息”继续发送
6. assistant 输出按增量写入本地，异常时保留为 `partial`
7. 用户切换 provider、账号、升级客户端后，线程列表和消息都从本地恢复

### 7.1.1 UX Expectations

- 用户感知到的线程应始终是同一个本地线程，不因 provider 或账号切换而生成新的本地线程
- 当系统触发摘要续聊时，需在产品层提供明确但轻量的提示，避免用户误以为上下文被完整直传
- 当线程只能显示但无法继续对话时，需给出明确原因，不允许静默失败

### 7.1.2 User Usage Model

- 默认情况下，该功能应尽量无感。用户不需要理解 `response_id`、provider 链路或本地重建细节，只需要像平时一样继续原线程
- 用户切换 provider、切换账号或升级桌面端后，应直接进入原线程继续发送下一条消息，不需要手动迁移或重新建立线程
- 当上下文过长时，系统自动触发摘要续聊，并给出轻提示，例如“已压缩前文以继续对话”
- 当线程可显示但不可继续对话时，系统需明确说明原因，并提供下一步动作，例如新开线程或导出备份
- 当用户需要掌控数据时，可以手动使用导出 / 恢复能力完成备份、迁移或排查

### 7.2 Key Features

1. 独立本地线程存储
2. 标准化消息模型
3. provider / 账号路由记录
4. `previous_response_id` 可选复用
5. 流式中断恢复
6. 超长上下文摘要续聊
7. JSONL 导出与恢复
8. schema migration
9. 用户可见的 provider / 账号标记

### 7.3 Thread Identity Model

- 线程在本地是唯一资产，`thread_id` 不随 provider、账号、模型切换而变化
- provider / account / model 作为线程上的 route 历史记录保存，不参与定义线程身份
- 同一线程下允许存在多次 route 切换记录，用于恢复和问题排查
- 用户在 UI 中看到的是本地线程视图，而不是某个 provider 的远端线程视图

### 7.4 Continuation Strategy

- 优先复用远端 `previous_response_id`，仅作为性能优化，不作为续聊前提
- 远端链路不可复用时，使用本地 canonical transcript 直接重建上下文
- 当 transcript 超出上下文窗口时，先生成本地摘要，再按 token 预算拼接最近消息继续对话，而不是按固定轮数截断
- 摘要优先由当前目标模型生成；若摘要生成失败，则降级为截断后的短上下文续聊，并向用户明确提示
- 本期仅承诺文本线程和长上下文文本线程的续聊能力
- 对 tool call、多模态等不在本期范围内的线程，系统可以显示历史，但不承诺跨 provider 续聊成功

### 7.5 Data Boundary

- 本地存储采用明文 SQLite
- 导出采用明文 JSONL，默认包含完整消息和基础元数据
- 本期存储范围包括线程、消息、消息状态、时间戳、provider、account、model、route 历史、摘要结果
- 本期不做本地加密、不做导出脱敏、不做云端同步
- 用户需可见导出和恢复入口，但隐私增强能力不纳入本期

### 7.6 Technology

- 本地主存储：SQLite
- 备份格式：JSONL
- 消息模型：canonical transcript
- 路由模型：provider + account + model + capabilities
- 恢复方式：优先本地 transcript，远端状态仅作优化

### 7.7 Assumptions

- Codex 桌面端允许项目接入独立的本地数据目录
- 不同第三方 Responses API 至少能完成基础文本响应
- provider 兼容程度不一致，因此必须支持自动降级
- 摘要续聊只针对文本上下文，不要求保留 tool execution 级别的完整可重放语义

## 8. Release

### Phase 1

- 建立 SQLite 基础表
- 建立本地线程和消息存储
- 接入最小可用的 Responses API 发送层
- 实现切 provider / 账号时的本地上下文重建
- 实现超长上下文的摘要续聊
- 实现流式增量写盘与 `partial` / `failed` 状态保留
- 提供用户可见的最小完整闭环：线程可见、消息可恢复、切换后可继续对话

### Phase 2

- 接入 checkpoint 和 route 记录
- 接入自动备份和恢复
- 接入导出 / 导入
- 补齐基础异常提示和恢复提示

### Phase 3

- 接入会话列表、搜索、收藏、归档
- 接入 UI 路由提示和隐私控制

## 9. Acceptance Criteria

### 9.1 Success Definition

- “显示成功”指线程列表可见、标题可见、历史消息可读
- “继续对话成功”指用户发送一条新消息后，系统能基于该线程已有本地上下文拿到新的 assistant 回复并写回本地
- “长上下文线程”指完整 transcript 已超过当前目标模型上下文窗口，必须经过摘要才能继续对话的文本线程

### 9.2 Acceptance Scenarios

| 场景 | 输入条件 | 期望结果 |
| --- | --- | --- |
| 切换 provider | 同一新线程切换到另一个 provider | 原线程仍显示，历史消息完整可读，可继续发送新消息 |
| 切换账号 | 同一新线程切换到另一个账号 | 原线程仍显示，历史消息完整可读，可继续发送新消息 |
| 远端链路失效 | 删除或无法使用 `previous_response_id` | 系统回退到本地 transcript 重建，线程仍可继续对话 |
| 长上下文续聊 | transcript 超出模型上下文窗口 | 系统先摘要，再继续对话，并保留摘要续聊标记 |
| 流式中断 | 网络中断或进程终止 | 最后一轮保留为 `partial` 或 `failed`，不出现整段消失 |
| 客户端升级 | 升级桌面端后重新打开应用 | 新方案线程可恢复显示，消息可读，可继续对话 |

### 9.3 Phase 1 Exit Criteria

- 用户可以在 UI 中看到由新方案创建的线程和历史消息
- 用户切换 provider 或账号后，仍能在同一线程中继续文本对话
- 长上下文文本线程可以通过摘要继续对话
- 流式中断不会导致最后一轮消息整段消失

## 10. Design Inputs

### 10.1 Minimum Data Model

以下为支持本期目标的最小数据对象，字段名可在技术方案阶段调整，但对象职责不建议再缩减。

| 对象 | 最小必需字段 | 说明 |
| --- | --- | --- |
| `threads` | `thread_id`、`title`、`created_at`、`updated_at`、`status` | 本地线程主表，`thread_id` 是用户感知上的唯一线程身份 |
| `messages` | `message_id`、`thread_id`、`seq`、`role`、`content`、`status`、`created_at`、`updated_at` | 保存 canonical transcript；`status` 用于流式和异常恢复 |
| `routes` | `route_id`、`thread_id`、`provider`、`account_id`、`model`、`previous_response_id`、`continuation_mode`、`created_at` | 记录线程在不同 provider / 账号 / 模型上的续聊路径 |
| `summaries` | `summary_id`、`thread_id`、`source_range`、`summary_text`、`status`、`model`、`created_at` | 保存长上下文压缩结果，供摘要续聊和排查使用 |

补充约束：
- `threads` 与 `messages` 是本期绝对核心；没有这两张表就无法满足恢复和续聊要求
- `routes` 必须保留历史记录，不能只覆盖最新值，否则无法解释线程为何在某次切换后改变续聊策略
- `summaries` 可以独立成表，也可以在实现中折叠到其他表，但摘要结果必须可追踪

### 10.2 Message State Machine

本期至少需要覆盖 assistant 消息的以下状态：

| 状态 | 含义 | 允许流转 |
| --- | --- | --- |
| `pending` | 本地已创建占位，远端请求尚未开始或尚未收到内容 | `streaming`、`failed` |
| `streaming` | 正在接收增量输出并持续写盘 | `completed`、`partial`、`failed` |
| `completed` | 正常完成，内容可作为后续续聊输入 | 终态 |
| `partial` | 已收到部分内容，但因中断/崩溃/断网未完整结束 | 终态，可在后续重试时继续参考 |
| `failed` | 请求失败，未形成可用完整回复 | 终态 |

补充约束：
- 用户消息在发送前必须先落本地，至少要保证“用户已输入的内容不丢”
- assistant 增量内容必须边接收边写盘，不能只在流结束时一次性写入
- `partial` 与 `failed` 必须在 UI 和导出中可见，不能被静默吞掉

### 10.3 Summary Continuation Decision

建议按如下顺序决策续聊路径：

1. 若当前 route 可安全复用远端 `previous_response_id`，优先走远端链路
2. 若远端不可复用，则评估本地 transcript 是否能在当前模型输入预算内直接重建
3. 若超出预算，则对较早消息做摘要压缩，并保留最近消息窗口
4. 若摘要成功且压缩后可落入预算，则继续发送
5. 若摘要失败但截断后的短上下文仍可发送，则降级发送，并提示用户已缩短上下文
6. 若降级后仍无法发送，则中止续聊，明确提示用户原因和下一步动作

实现约束：
- “最近消息窗口”按 token 预算保留，而不是按固定轮数保留
- token 预算需要预留系统指令、路由元数据和安全余量
- 具体阈值与安全余量由技术方案结合目标模型能力确定，当前记为 `TBD`

### 10.4 Minimum UI Touchpoints

为保证用户真正“用得上”这个能力，本期最少需要以下交互点：

| 位置 | 最小要求 | 用户价值 |
| --- | --- | --- |
| 线程列表 | 能稳定展示本地线程，不因 provider / 账号切换消失 | 用户能找回原工作线程 |
| 线程详情页 | 能看到完整历史消息及最后一轮状态 | 用户能理解线程目前停在什么位置 |
| 摘要续聊提示 | 当系统触发摘要或截断降级时，展示轻提示 | 用户知道系统为继续对话做了什么 |
| 失败提示 | 当线程可显示但不可继续对话时，展示明确原因和下一步动作 | 用户不会误以为产品无响应 |
| 导出 / 恢复入口 | 至少有一个用户可达入口 | 用户可以主动备份、迁移、排查 |

建议的提示文案方向：
- 摘要续聊：`已压缩前文以继续对话`
- 降级续聊：`上下文过长，系统已缩短历史后继续`
- 无法续聊：`当前线程可查看，但暂时无法继续对话`

### 10.5 Recommended Task Split

建议研发拆分为以下最小任务单元：

1. 本地线程与消息存储层
2. 流式写盘与消息状态管理
3. route 记录与切换后的续聊策略
4. 长上下文摘要与 token 预算裁剪
5. 线程列表 / 详情页读取本地数据
6. 异常提示、摘要提示、导出恢复入口

### 10.6 TBDs For Technical Design

以下内容建议在技术方案中继续拍板，本 PRD 暂不强行定死：

- `account_id` 的本地标识形式，是否直接复用现有账号主键
- token 预算计算方式，以及不同模型的安全余量
- 摘要结果是否需要版本化，以支持后续重复摘要或替换摘要
- 导出 / 恢复入口放在线程页、设置页，还是两者同时提供
