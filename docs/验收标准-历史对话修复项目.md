# 验收标准：Codex 桌面端历史对话修复项目

## 1. 验收目标

本验收文档用于判断项目是否真正解决“历史对话丢失或无法继续对话”问题。验收以用户可感知结果为准，不以“底层可能还有残留数据”作为通过依据。

本项目的验收目标是确认：新方案上线后创建的新文本线程，在切换 provider、切换账号、升级桌面端、流式中断、远端链路失效、长上下文超窗等场景下，仍然能够稳定显示、尽可能继续对话，并保留必要的异常状态和用户提示。

## 2. 验收范围与口径

### 2.1 本期验收范围

本期只验收以下范围：
- 新方案上线后创建的新线程
- 文本对话线程
- 长上下文文本线程的摘要续聊
- 本地 SQLite 恢复
- JSONL 导出与导入恢复
- provider / 账号 / 模型切换后的同线程续聊

### 2.2 不在本期验收范围内

以下内容不作为本期阻塞项：
- 新方案上线前历史线程的迁移、回填、修复
- tool call 线程的跨 provider 兼容续聊
- 图片和其他多模态消息的跨 provider 兼容续聊
- 云端同步和多设备同步
- 搜索、收藏、归档等后续阶段会话管理增强能力

### 2.3 统一口径

- 显示成功：线程列表可见、标题可见、历史消息可读
- 继续对话成功：用户发送新消息后，系统能基于该线程已有本地上下文获得新的 assistant 回复并写回本地
- 长上下文线程：完整 transcript 超出目标模型上下文窗口，必须经过摘要后才能继续对话的文本线程
- 远端链路失效：`previous_response_id` 或远端上下文链路不可用，但本地 transcript 仍存在
- 最后一轮不丢失：流式中断后，最后一轮必须保留为 `partial` 或 `failed`，且已落盘文本仍可见

### 2.4 分层验收口径

结合当前仓库实现，验收必须拆成两层进行，避免“底层原型尚未接入桌面端，却直接按完整产品形态验收”。

当前阶段性状态：仓库已具备 Codex.app 扩展历史补丁和底层 engine 雏形。2026-04-25 已在真实 Codex.app 中完成当前主路径验证：当前项目的历史列表不再只显示当前 provider 的 3 条，`custom / sub2api / openai` 历史已可同时显示。仍建议继续补测更换登录账号、打开旧对话继续发送、升级后 LaunchAgent 自动重打补丁等场景。

第一层：底层引擎验收
- 目标是验证 SQLite 持久化、续聊策略、异常恢复、导出导入是否成立
- 允许通过单元测试、集成测试、CLI 或脚本方式执行
- 该层通过，只能证明“引擎能力成立”

第二层：桌面端集成验收
- 目标是验证桌面端是否真正接通本地线程读取、发送链路、启动恢复、升级迁移和提示交互
- 必须在真实桌面端产品链路中执行
- 只有该层通过，才能判定“用户侧功能成立”

## 3. 核心验收原则

### 3.1 本地历史优先

- UI 展示线程和消息时，不依赖远端 provider 历史查询
- 线程是否显示，以本地存储结果为准

### 3.2 线程身份稳定

- 同一线程在切换 provider、账号、模型后，仍应保持同一个本地线程身份
- 不允许因路由切换而生成新的本地线程来“伪装通过”

### 3.3 失败可见

- 线程可显示但无法继续对话时，必须给出明确原因
- 不允许出现静默失败、空白状态、无提示回退

### 3.4 数据可恢复

- 异常中断、应用崩溃、升级迁移失败时，至少要保证历史可见
- 不允许最后一轮消息整段消失

### 3.5 分层结论不能混用

- 引擎层通过，不等于桌面端已经接入
- 桌面端验收失败时，不允许用“底层库测试通过”替代产品通过结论
- 桌面端上线判断必须以桌面端集成验收结果为准

## 4. 核心验收项

### 4.0 验收分阶段要求

执行顺序必须为：
1. 先完成底层引擎验收
2. 再完成桌面端集成验收

若底层引擎未通过，不进入桌面端集成验收；若桌面端尚未接入，则只能输出“引擎通过 / 桌面端未验”的阶段性结论。

### 4.1 历史显示

必须满足：
- 新方案创建的线程在重启应用后仍可见
- 线程标题可恢复
- 历史消息正文完整可读
- 消息顺序与发送顺序一致
- 线程最近更新时间与最后消息时间合理更新

不要求本期验收：
- 搜索命中
- 收藏状态恢复
- 归档状态恢复

### 4.2 历史续聊

必须满足：
- 同一 provider / 同一账号下，线程可继续对话
- 切换 provider 后，线程仍可继续对话
- 切换账号后，线程仍可继续对话
- 删除或失去远端 `response_id` 后，线程仍可尽可能继续对话
- 切换后仍使用同一个本地线程，而不是新建线程

### 4.3 长上下文摘要续聊

必须满足：
- 当 transcript 超出模型上下文窗口时，系统能够触发摘要续聊
- 摘要续聊成功时，用户仍可在原线程继续发送并获得新回复
- 触发摘要续聊时，UI 有轻量提示
- 摘要结果可追踪，便于排查

允许的降级：
- 摘要生成失败时，可退化为更短窗口续聊
- 发生降级时，必须给出明确提示

### 4.4 异常恢复

必须满足：
- 流式中断后，最后一轮保留为 `partial` 或 `failed`
- 请求失败后，最后一轮状态准确，不整段消失
- 应用崩溃后，重新启动仍可恢复线程
- 断网恢复后，历史不消失
- 启动恢复时，残留 `streaming` 状态会被正确收敛

### 4.5 升级恢复

必须满足：
- 升级桌面端后，线程列表仍可恢复
- 历史消息正文仍可恢复
- 路由信息仍可用于后续续聊判断
- migration 失败时，至少可进入只读恢复模式，确保历史可读

### 4.6 导出与恢复

必须满足：
- 用户可以导出新方案线程
- 导出文件包含恢复所需的线程、消息和基础元数据
- 在清空或替换本地数据库后，可通过 JSONL 恢复历史
- 恢复后的线程可显示，且至少文本线程可继续对话

## 5. 验收优先级

### 5.1 P0 阻塞项

以下任一失败都视为项目不通过：
- 切换 provider 后线程消失
- 切换账号后线程消失
- 线程可见但无法继续对话且没有明确原因
- 删除远端 `response_id` 后无法回退到本地 transcript 重建
- 流式中断后最后一轮整段消失
- 升级后线程列表无法恢复
- 长上下文线程无法通过摘要或降级策略继续对话

### 5.2 P1 重要项

以下问题不一定阻塞上线，但必须记录并评估：
- provider / 账号标识展示不清晰
- 摘要续聊提示不明显
- 导入恢复后需要人工刷新才能看到线程
- 错误提示存在但表述不够明确

## 6. 验收环境要求

执行验收时，至少覆盖以下环境组合：
- 官方账号场景
- 第三方 provider 场景
- 同一线程跨 provider 场景
- 网络正常与网络中断场景
- 升级前版本到升级后版本场景

建议准备至少两类测试线程：
- 普通文本线程：3 至 5 轮对话
- 长上下文线程：足以触发摘要续聊

补充要求：
- 底层引擎验收可使用独立数据库和 mock provider 执行
- 桌面端集成验收必须在真实桌面端构建中执行，不以 CLI 结果替代
- 桌面端集成验收需覆盖“应用启动自动恢复”而非人工执行 `recover`

## 7. 验收用例

### Case 1：底层引擎基础续聊

步骤：
1. 使用引擎接口或 CLI 创建一个新线程
2. 完成 3 轮文本对话
3. 重新打开同一数据库
4. 在同一线程继续追问

预期：
- 线程和消息可从本地数据库读取
- 新消息成功生成
- 未新建额外本地线程

### Case 2：底层引擎切换第三方 provider

步骤：
1. 使用 provider A 创建线程并完成 2 轮对话
2. 切换到 provider B
3. 在同一线程继续追问

预期：
- 旧消息仍完整可读
- 系统继续使用原本地线程
- 新消息成功生成
- 即使未复用远端链路，也不影响续聊结果

### Case 3：底层引擎删除远端 checkpoint

步骤：
1. 创建一个新线程并完成多轮对话
2. 删除该线程关联的 `previous_response_id` / `remote_response_id`
3. 在同一线程继续发送消息

预期：
- 历史显示不受影响
- 系统回退到本地 transcript 重建
- 新消息成功生成

### Case 4：底层引擎长上下文摘要续聊

步骤：
1. 创建一个长上下文文本线程，持续对话直到超过目标模型上下文窗口
2. 在同一线程继续发送新消息

预期：
- 系统触发摘要续聊或等价降级路径
- 新回复成功生成
- 线程仍然是原线程
- 摘要结果可追踪

### Case 5：底层引擎流式中断与恢复

步骤：
1. 发起一次流式请求
2. 在输出过程中人为断网、终止请求或中断进程
3. 重新打开数据库并执行恢复逻辑

预期：
- 最后一轮消息保留为 `partial` 或 `failed`
- 已生成文本仍可见
- 不出现最后一轮整段消失

### Case 6：底层引擎导出与恢复

步骤：
1. 导出新方案线程
2. 清空或替换本地数据库
3. 使用导出文件执行恢复
4. 在恢复后的线程中继续对话

预期：
- 线程、标题、消息、基础路由信息恢复
- 恢复后的消息顺序正确
- 至少文本线程可以继续对话

### Case 7：桌面端同 provider 同账号基础续聊

步骤：
1. 在 Codex 桌面端创建一个新线程
2. 完成 3 轮文本对话
3. 关闭并重开 Codex 桌面端
4. 在同一线程继续追问

预期：
- 线程仍显示在列表中
- 标题和历史消息完整可见
- 新回复成功生成
- 未新建额外本地线程

### Case 8：桌面端切换第三方 provider

步骤：
1. 使用 provider A 创建线程并完成 2 轮对话
2. 切换到 provider B
3. 在同一线程继续提问

预期：
- 旧消息仍完整可读
- 系统继续使用原本地线程
- 新消息成功生成
- UI 可见 provider 已切换
- 若未复用远端链路，也不影响续聊结果

### Case 9：桌面端切换官方账号

步骤：
1. 使用官方账号 A 创建线程并完成 2 轮对话
2. 切换到官方账号 B
3. 在同一线程继续提问

预期：
- 历史仍完整可见
- 系统不创建新本地线程
- 新回复成功生成
- 若走本地 transcript 重建，上层表现仍然连续

### Case 10：桌面端长上下文摘要续聊

步骤：
1. 创建一个长上下文文本线程，持续对话直到超过目标模型上下文窗口
2. 在同一线程继续发送新消息

预期：
- 系统触发摘要续聊或等价降级路径
- 用户可看到轻量提示
- 新回复成功生成
- 线程仍然是原线程

### Case 11：桌面端流式中断

步骤：
1. 发起一次流式请求
2. 在输出过程中人为断网或终止请求
3. 重新进入线程

预期：
- 最后一轮消息保留为 `partial` 或 `failed`
- 已生成文本仍可见
- 不出现最后一轮整段消失

### Case 12：桌面端应用崩溃恢复

步骤：
1. 发起一次流式请求
2. 在输出过程中强制关闭客户端进程
3. 重启应用并进入原线程

预期：
- 线程仍可见
- 最后一轮消息状态被正确收敛
- 已写入本地的文本仍可见

### Case 13：桌面端升级

步骤：
1. 创建多条新方案线程
2. 升级 Codex 桌面端版本
3. 重启应用
4. 随机抽取线程继续对话

预期：
- 线程列表仍可恢复
- 消息内容仍可恢复
- 抽取线程可继续对话
- 若 migration 失败，仍至少可只读查看历史

### Case 14：桌面端导出与恢复

步骤：
1. 导出新方案线程
2. 清空或替换本地数据库
3. 使用导出文件执行恢复
4. 在恢复后的线程中继续对话

预期：
- 线程、标题、消息、基础路由信息恢复
- 恢复后的消息顺序正确
- 至少文本线程可以继续对话

### Case 15：线程可见但不可续聊

步骤：
1. 构造一个无法继续对话的异常条件，例如目标 provider 不可用且本地恢复条件不足
2. 打开原线程并尝试继续发送

预期：
- 线程仍可见，历史仍可读
- 系统明确说明无法继续对话的原因
- 不出现静默失败或无反馈状态

## 8. 量化通过标准

满足以下条件才可判定通过：
- 底层引擎 P0 用例全部通过
- 若目标是桌面端上线，则桌面端 P0 用例也必须全部通过
- 桌面端集成阶段需满足以下 KR：
- KR1：新方案文本线程在切换 provider 后，100% 可显示
- KR2：新方案文本线程在切换账号后，100% 可显示且可继续对话
- KR3：删除所有远端 `response_id` 后，至少 95% 的线程仍可继续对话
- KR4：桌面端升级后，新方案线程恢复成功率达到 99%
- KR5：流式中断后，100% 的最后一轮消息保留为 `partial` 或 `failed`
- KR6：长上下文文本线程的摘要续聊成功率不低于 90%

补充要求：
- 所有成功率统计都仅针对“新方案上线后创建的新文本线程”
- 若样本量不足，应先补足测试样本，再判定是否通过
- 若当前阶段尚未接入桌面端，则只能输出“底层引擎通过”，不得提前给出桌面端通过结论

## 9. 不通过判定

出现以下任一情况，即判定为不通过：
- 任意新方案线程在切换 provider 后消失
- 任意新方案线程在切换账号后消失
- 能看到历史但发送新消息后无结果、无错误、无提示
- 删除远端链路后，线程无法回退到本地 transcript 重建
- 长上下文线程在超窗后既不能摘要续聊，也没有明确降级结果
- 流式中断后最后一轮整段消失
- 恢复后消息顺序错乱或内容大面积缺失
- 升级后线程列表不可恢复，且没有只读兜底
- 桌面端尚未接入本地线程读取，却以 CLI 结果宣称桌面端通过

## 10. 验收结论模板

验收结论建议按以下格式输出：

| 项目 | 结果 | 说明 |
| --- | --- | --- |
| 历史显示 | 通过 / 不通过 | 记录问题摘要 |
| 历史续聊 | 通过 / 不通过 | 记录问题摘要 |
| 长上下文摘要续聊 | 通过 / 不通过 | 记录问题摘要 |
| 异常恢复 | 通过 / 不通过 | 记录问题摘要 |
| 升级恢复 | 通过 / 不通过 | 记录问题摘要 |
| 导出与恢复 | 通过 / 不通过 | 记录问题摘要 |
| 总结论 | 通过 / 不通过 | 是否满足上线条件 |
