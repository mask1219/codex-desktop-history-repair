# 技术方案：Codex 桌面端历史对话修复项目

## 1. 目标与边界

### 1.1 设计目标

本方案用于支撑最新版 PRD 中定义的目标：为新方案上线后创建的文本线程建立一套独立于上游 provider 和 Codex 内部存储的本地会话系统，使线程在以下场景中仍然可见、可恢复、可继续对话：
- 切换第三方 provider
- 切换官方账号
- 升级 Codex 桌面端
- 断网、崩溃、流式中断
- 长上下文超出模型窗口后继续对话

### 1.2 本期范围

本期只覆盖以下对象和能力：
- 新方案上线后创建的新线程
- 文本对话线程
- 长上下文文本线程的摘要续聊
- 本地线程恢复、导出、导入恢复
- provider / 账号 / 模型切换后的同线程续聊

本期明确不覆盖：
- 历史存量线程迁移修复
- tool call 线程的跨 provider 兼容续聊
- 图片和其他多模态消息的跨 provider 兼容续聊
- 云端同步和多设备同步

### 1.3 成功定义

方案实现需直接支撑以下产品验收定义：
- 显示成功：线程列表可见、标题可见、历史消息可读
- 继续对话成功：用户发送新消息后，系统能基于该线程已有本地上下文拿到新的 assistant 回复并写回本地
- 长上下文线程：完整 transcript 超出目标模型上下文窗口，必须经过摘要后才能继续对话

### 1.4 当前实现状态

结合当前仓库实现，本项目现阶段应被定义为“Codex.app 扩展历史补丁 + 历史修复底层引擎雏形”，而不是另起一套独立会话服务。

当前已经具备：
- SQLite schema 初始化与基础 migration
- 本地 `threads / messages / routes / summaries` 持久化
- `remote_chain / local_rebuild / summary_rebuild` 续聊策略
- 流式增量写盘与 `partial / failed` 收口
- JSONL 导出 / 导入恢复
- CLI 入口：`init / recover / export / import / list / show / send`
- 宿主接线与 UI 数据适配雏形：`DesktopSessionHost`、`DesktopHistoryAdapter`
- Codex.app 打包补丁：普通新线程和恢复线程默认 `persistExtendedHistory: true`
- Codex.app 打包补丁：最近线程列表、项目入口最近线程列表、全部线程列表都已改为优先走 state db
- 已通过真实 app-server 调试确认：`thread/list` 只有在 `modelProviders: []` 时才会返回全部 provider 历史；`modelProviders: null` 只会返回当前 provider
- 2026-04-25 已在真实 Codex.app 中验证：当前项目历史列表可同时显示 `custom / sub2api / openai`

当前仍需验收或补齐：
- 真实 Codex.app 内更换登录账号后历史列表仍可见
- 真实 Codex.app 打开旧对话后可以继续发送消息
- Codex.app 更新后 LaunchAgent 能自动重打补丁
- provider / 账号切换、桌面端升级等量化 KR 的样本化验收

因此，本技术方案需要同时覆盖两件事：
- 底层引擎本身如何稳定工作
- 这套引擎如何以最小代价接入 Codex 桌面端

## 2. 核心原则

### 2.1 本地 canonical transcript 是唯一真相

- 本地 `threads` + `messages` 组成唯一可信的线程资产
- 远端 `response_id`、provider 会话、账号状态都不能作为线程主身份
- UI 永远从本地读取线程和消息，不以远端返回结果决定是否展示历史

### 2.2 远端链路只是优化，不是续聊前提

- `previous_response_id` 仅在兼容且稳定时复用，用于减少重传成本
- 远端链路不可复用时，必须自动回退为“本地 transcript 重建上下文”
- 删除远端 `response_id` 后，线程仍需尽可能继续对话

### 2.3 线程身份稳定，路由历史可追踪

- `thread_id` 是用户感知上的唯一线程身份
- provider / account / model 仅作为 route 历史记录保存
- 切换 provider、账号、模型时不创建新的本地线程

### 2.4 失败要可见，可恢复，不静默丢数据

- assistant 消息在流式过程中增量落盘
- 中断时保留 `partial` 或 `failed` 状态
- 不允许最后一轮消息整段消失
- 线程可显示但不可继续时，必须能给出明确原因

### 2.5 方案尽量最小闭环

本期只保留完成 PRD 所需的最小能力，不引入与当前目标无关的分支、多人协作、云同步、复杂隐私增强或多模态抽象。

## 3. 总体架构

### 3.1 模块划分

- `ThreadRepository`：管理线程元数据、标题、状态、时间戳
- `MessageRepository`：管理 canonical transcript、消息顺序、消息状态
- `RouteRepository`：记录每次发送所使用的 provider / account / model / continuation mode
- `ContinuationPlanner`：决定本轮使用远端链路复用、本地 transcript 重建，还是摘要续聊
- `SummaryManager`：负责长上下文压缩、摘要落盘、摘要可追踪
- `StreamWriter`：负责流式增量写盘、完成态收口、异常态保留
- `ExportImportService`：负责 JSONL 导出与导入恢复
- `MigrationManager`：负责 schema version 管理、升级迁移、失败回滚

### 3.2 当前代码形态

当前仓库中的能力主要集中在底层引擎层：
- 数据层：`SessionDatabase`、各类 Repository、导出导入逻辑
- 续聊层：`ContinuationPlanner`、`SummaryManager`、`ChatSessionService`
- provider 层：`ResponsesApiProviderClient`
- 调试与验证入口：CLI

这意味着现有代码已经可以支撑本地存储、续聊策略和异常恢复验证，但还不能独立证明桌面端产品链路已经打通。

### 3.3 存储分层

- 主存储：SQLite，保存线程、消息、路由、摘要、schema version
- 导出格式：JSONL，作为手动备份和恢复载体
- 远端状态：仅保存最小续聊优化信息，不作为恢复前提

### 3.4 桌面端最小接入架构

接入 Codex 桌面端时，建议按以下最小分层实现：
- 桌面端宿主层：负责应用启动、账号切换、provider 切换、升级迁移时机控制
- 本地会话服务层：复用本仓库中的 `ChatSessionService`、Repository、恢复与导出导入能力
- UI 数据适配层：把本地 `threads` 和 `messages` 转换成线程列表页、详情页所需视图模型
- UI 交互层：展示线程列表、线程详情、摘要提示、失败提示、导出恢复入口

桌面端宿主最少需要接入以下调用点：
1. 应用启动时执行 migration 和 `recover`
2. 打开线程列表时从本地 SQLite 读取线程
3. 打开线程详情时从本地 SQLite 读取消息
4. 用户发送消息时调用本地会话服务，而不是只依赖远端会话链路
5. 切换 provider / 账号时保持同一个本地 `thread_id`
6. 导出 / 恢复入口调用本地导出导入服务

### 3.5 高层数据流

1. 用户发送消息，先写入本地 `messages`
2. 系统创建 assistant 占位消息
3. `ContinuationPlanner` 根据当前 route 和 transcript 状态决定续聊方式
4. 若上下文过长，则先生成摘要并记录到 `summaries`
5. 请求发往当前 provider，流式输出由 `StreamWriter` 增量写盘
6. 请求完成后写入最终消息状态和 route 历史
7. 桌面端重启、升级或切换账号后，UI 直接从本地 SQLite 恢复线程

## 4. 数据模型

### 4.1 表设计概览

本期采用以下最小数据对象：
- `threads`
- `messages`
- `routes`
- `summaries`
- `app_meta`

其中 `threads`、`messages`、`routes`、`summaries` 对齐 PRD 最小数据模型；`app_meta` 仅用于 schema version 和本地元信息管理。

### 4.2 threads

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `thread_id` | text pk | 本地线程唯一身份 |
| `title` | text | 线程标题 |
| `status` | text | `active` / `archived` / `deleted` |
| `created_at` | integer | 创建时间 |
| `updated_at` | integer | 最近更新时间 |
| `last_message_at` | integer | 最近消息时间 |
| `last_continuation_error` | text nullable | 最近一次无法续聊的可展示原因 |

说明：
- `thread_id` 不随 provider / account / model 切换而变化
- 本期不引入 workspace 维度和复杂分支结构，避免超范围设计

### 4.3 messages

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `message_id` | text pk | 消息主键 |
| `thread_id` | text fk | 所属线程 |
| `seq` | integer | 线程内单调递增顺序 |
| `role` | text | `user` / `assistant` / `system` |
| `content` | text | 文本内容 |
| `status` | text | `pending` / `streaming` / `completed` / `partial` / `failed` / `canceled` |
| `created_at` | integer | 创建时间 |
| `updated_at` | integer | 更新时间 |
| `provider` | text nullable | 生成该消息时所用 provider |
| `account_id` | text nullable | 生成该消息时所用账号标识 |
| `model` | text nullable | 生成该消息时所用模型 |
| `request_id` | text nullable | 请求追踪 id |
| `error_code` | text nullable | 错误码 |
| `error_message` | text nullable | 错误摘要 |

说明：
- 本期 `content` 仅承载文本消息
- 不为 tool call、多模态 parts 预留抽象层，避免过度设计
- assistant 流式过程直接更新同一条消息记录，确保崩溃后仍能读到已落盘内容

### 4.4 routes

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `route_id` | text pk | 路由记录主键 |
| `thread_id` | text fk | 所属线程 |
| `provider` | text | 当前 provider |
| `account_id` | text | 当前账号标识 |
| `model` | text | 当前模型 |
| `previous_response_id` | text nullable | 本轮发送前可复用的远端 checkpoint |
| `remote_response_id` | text nullable | 本轮成功后的远端 response id |
| `continuation_mode` | text | `remote_chain` / `local_rebuild` / `summary_rebuild` |
| `capabilities_snapshot` | text nullable | 兼容性快照，建议 JSON |
| `created_at` | integer | 创建时间 |

说明：
- `routes` 记录历史，不覆盖旧值，便于解释线程为何在某次切换后改变续聊策略
- `previous_response_id` 和 `remote_response_id` 只承担优化和排查职责

### 4.5 summaries

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `summary_id` | text pk | 摘要记录主键 |
| `thread_id` | text fk | 所属线程 |
| `source_start_seq` | integer | 摘要覆盖的起始消息序号 |
| `source_end_seq` | integer | 摘要覆盖的结束消息序号 |
| `summary_text` | text | 摘要内容 |
| `status` | text | `completed` / `failed` |
| `model` | text | 生成摘要所用模型 |
| `created_at` | integer | 创建时间 |

说明：
- 摘要必须可追踪，方便排查“为何继续对话时上下文与原文不完全一致”
- 本期只记录摘要结果，不做复杂摘要树或多版本管理

### 4.6 app_meta

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `key` | text pk | 元信息键 |
| `value` | text | 元信息值 |

最少保存：
- `schema_version`
- `data_format_version`

## 5. 续聊决策与请求流程

### 5.1 续聊模式

系统只允许三种续聊模式：
- `remote_chain`：复用远端 `previous_response_id`
- `local_rebuild`：用本地 transcript 重建上下文
- `summary_rebuild`：用“摘要 + 最近消息”重建上下文

三者优先级：
1. 优先尝试 `remote_chain`
2. 不满足条件时回退 `local_rebuild`
3. 若 `local_rebuild` 超出上下文预算，则进入 `summary_rebuild`

### 5.2 remote_chain 复用条件

仅当以下条件全部满足时才允许复用远端链路：
- 当前线程存在最近一次成功的 `remote_response_id`
- provider 未切换
- account 未切换
- 当前目标模型与上一轮兼容，且明确支持 `previous_response_id`
- 最近一次远端续链没有失败记录
- 兼容性探测结果仍然有效

任一条件不满足，都直接放弃复用，不进行脆弱重试。

### 5.3 local_rebuild 规则

触发条件：
- provider / account / model 任一发生切换
- 当前 provider 不支持 `previous_response_id`
- 上一次远端续链失败
- 本地明确要求不依赖远端状态

执行规则：
1. 从本地读取该线程的 canonical transcript
2. 按消息顺序构建 `input`
3. 必要时附加稳定的 `instructions`
4. 发起新的远端请求，不引用旧 checkpoint

### 5.4 summary_rebuild 规则

触发条件：
- 完整 transcript 超出当前目标模型上下文窗口

执行规则：
1. 基于 token 预算评估可直接发送的消息量
2. 将较早消息压缩为摘要，摘要覆盖范围写入 `summaries`
3. 用“摘要 + 最近若干条原始消息 + 当前用户消息”构建请求
4. 记录本轮 `continuation_mode = summary_rebuild`

降级规则：
- 摘要优先由当前目标模型生成
- 若摘要生成失败，则降级为更短窗口的 `local_rebuild`
- 发生降级时，线程需记录可展示提示，例如“已压缩前文失败，已按最近消息继续对话”

### 5.5 单次发送完整流程

1. 写入 user message，状态直接为 `completed`
2. 创建 assistant 占位消息，状态为 `pending`
3. 读取当前 route 历史和 transcript 长度
4. 由 `ContinuationPlanner` 选择续聊模式
5. 若需摘要，则先生成并持久化摘要记录
6. 发起远端流式请求
7. 首个 token 到达后将 assistant 消息状态改为 `streaming`
8. 每次增量输出都更新 `content` 和 `updated_at`
9. 正常结束后将 assistant 消息状态改为 `completed`
10. 写入本轮 `routes`
11. 更新 `threads.updated_at` 和 `last_message_at`

## 6. 流式落盘与异常恢复

### 6.1 消息状态机

assistant 消息状态定义如下：
- `pending`：占位消息已创建，尚未收到流
- `streaming`：已收到部分输出，仍在持续写入
- `completed`：流正常结束
- `partial`：流异常结束，但已收到部分输出
- `failed`：请求失败，且没有有效输出
- `canceled`：用户主动中止

状态流转：
1. `pending -> streaming`
2. `streaming -> completed`
3. `streaming -> partial`
4. `pending -> failed`
5. `streaming -> canceled`

### 6.2 落盘策略

- assistant 占位消息必须在请求发出前落盘
- 流式增量按块更新同一条消息记录，不依赖内存缓冲到结束后一次性写入
- 每次状态变化都更新 `updated_at`
- 远端返回成功后再补写 `remote_response_id`

### 6.3 崩溃恢复

应用启动时执行一次恢复扫描：
- 所有 `streaming` 状态消息统一收敛为 `partial`
- 所有长时间停留在 `pending` 且无输出的 assistant 消息收敛为 `failed`
- 恢复后线程仍需可见，且保留最后已写入的文本

该策略直接对应 PRD 中“流式中断后最后一轮整段不能消失”的要求。

桌面端接入要求：
- 恢复扫描必须在应用启动早期自动执行，而不是要求用户手动触发
- 恢复结果必须被线程列表页和详情页直接消费，避免“数据已恢复但 UI 未刷新”
- 若恢复过程中发现不可修复异常，线程仍需保持可见，并暴露明确错误原因

## 7. 导出、恢复与升级

### 7.1 SQLite 作为主恢复源

- 正常情况下，线程恢复只依赖本地 SQLite
- 应用启动时直接从 `threads` 和 `messages` 构建列表和详情
- 不需要向远端请求历史才能展示线程

### 7.2 JSONL 导出格式

导出采用明文 JSONL，每行一个对象，最小包含：
- `thread`
- `message`
- `route`
- `summary`

导出原则：
- 默认导出完整消息文本和基础元数据
- 不导出 API Key 等密钥材料
- 导出格式追求可恢复和可排查，不追求事件溯源级复杂度

### 7.3 JSONL 导入恢复

导入恢复流程：
1. 解析 JSONL
2. 校验对象类型和必需字段
3. 按 `threads -> messages -> routes -> summaries` 顺序导入
4. 重建必要索引
5. 对同 `thread_id` 冲突采取“跳过或生成新 id”的显式策略，避免静默覆盖

本期建议默认采用“导入到新线程副本”的策略，风险更低，也更符合手动恢复预期。

### 7.4 升级迁移

迁移原则：
- schema version 单独管理
- 启动早期执行迁移，且必须在事务中完成
- 迁移前自动生成本地备份
- 迁移失败时，应用进入只读恢复模式，至少保证历史可见

最小版本演进建议：
- `v1`：`threads` / `messages` / `routes` / `summaries` / `app_meta`
- `v2`：补充索引和兼容性字段，不改变核心身份模型

当前实现与目标差距：
- 当前代码已有基础 `apply_migrations()`，但还没有独立的桌面端 `MigrationManager` 编排层
- “迁移前自动生成本地备份”和“迁移失败进入只读恢复模式”仍需在桌面端宿主层补齐

## 8. UI 和交互约束

虽然本文件以技术实现为主，但以下交互约束必须由方案支撑：
- 用户看到的始终是同一个本地线程，不因 provider / 账号切换出现新的本地线程
- 当系统触发摘要续聊时，需要有轻提示，例如“已压缩前文以继续对话”
- 当线程可显示但不可继续对话时，需展示明确原因，不允许静默失败
- 用户需要有导出和恢复入口，但本期不做加密、脱敏和云同步

## 9. 兼容性与风险处理

### 9.1 provider 兼容性差异

风险：
- 不同第三方 Responses API 对 `previous_response_id` 的支持不一致

处理：
- 将远端链路复用视为可选优化
- 每次 route 记录保存能力快照
- 一旦发现兼容性不稳定，后续直接走本地重建

### 9.2 长上下文摘要偏差

风险：
- 摘要续聊天然可能损失细节

处理：
- 摘要覆盖范围可追踪
- 摘要只在超出预算时触发
- 保留最近若干条原始消息，避免最新上下文被过度压缩
- UI 明示当前为摘要续聊

### 9.3 本地明文存储边界

本期遵循 PRD 的数据边界：
- SQLite 为明文
- JSONL 导出为明文
- 不做本地加密
- 不做导出脱敏

为避免与既有账号系统耦合，本方案只在会话库中保存 `account_id` 等引用信息，不重复保存密钥材料。

## 10. 实施拆分

### 10.1 Phase 0：底层引擎原型

当前仓库已经基本完成以下内容：
- 建立 SQLite 基础表和 schema version
- 接入线程与消息写盘
- 完成 `remote_chain` / `local_rebuild` / `summary_rebuild`
- 完成流式增量写盘与 `partial` / `failed` 状态保留
- 提供 CLI 级别的 `send / list / show / recover / export / import`

本阶段目标是证明“引擎能力成立”，而不是证明“桌面端已经接通”。

### 10.2 Phase 1：桌面端最小接入闭环

- 在桌面端启动阶段接入 migration 和自动 `recover`
- 在线程列表页 / 详情页接入本地 SQLite 读取
- 在发送链路接入本地会话服务
- 打通应用重启、切 provider、切账号后的本地恢复与续聊闭环
- 保证 `partial / failed` 在 UI 可见

Phase 1 的上线判断标准：
- 用户在桌面端能够直接看到本地线程和消息
- 切换 provider / 账号后仍在同一线程继续发送
- 启动恢复无需用户手动操作

### 10.3 Phase 2：产品化补齐

- 补齐摘要续聊提示、失败提示、只读恢复提示
- 增加导出和导入恢复的用户可达入口
- 补齐兼容性快照和更明确的异常提示
- 优化摘要触发条件和恢复体验
- 增加迁移前备份和迁移失败只读恢复模式

### 10.4 非本期项

以下内容保留到后续版本再评估：
- 存量线程迁移
- tool call / 多模态跨 provider 续聊
- 云同步和多设备同步
- 搜索、收藏、归档等会话管理增强

## 11. 验收映射

| PRD 场景 | 技术方案支撑点 |
| --- | --- |
| 切换 provider | `thread_id` 不变，直接回退 `local_rebuild` 或 `summary_rebuild` |
| 切换账号 | UI 从本地 SQLite 展示历史，发送时重新选择 route |
| 远端链路失效 | 放弃 `previous_response_id`，使用 canonical transcript 重建 |
| 长上下文续聊 | `SummaryManager` 生成摘要并记录 `summaries` |
| 流式中断 | `StreamWriter` 增量落盘，启动恢复时收敛为 `partial` / `failed` |
| 客户端升级 | `MigrationManager` 事务化迁移，失败时至少保证只读可见 |

## 12. 结论

本方案将线程身份、历史展示和续聊能力统一收敛到本地 transcript 上，把 provider 会话从“唯一依赖”降级为“可选优化”。这样可以在不承诺存量迁移、不扩展到多模态和云同步的前提下，用最小实现闭环满足 PRD 的核心目标：新线程不丢、切换不丢、升级不丢、中断不丢，并且在长上下文下仍可继续对话。

需要特别明确的是：当前仓库已经证明“底层引擎方向成立”，下一阶段的核心工作不再是重新设计续聊策略，而是把这套引擎稳定接入桌面端宿主、UI 和升级链路，完成真正面向用户的产品闭环。
