# 桌面端接入与升级策略

## 当前推荐架构

当前核心需求是：继续使用 Codex.app 原生界面，在 `ccswitch` 切换第三方 API 或更换登录账号后，仍能在 App 内看到全部历史对话并继续沟通。

因此当前推荐路径是 Codex.app 打包补丁：

```bash
node scripts/patch_codex_app_extended_history.js /Applications/Codex.app --check
node scripts/patch_codex_app_extended_history.js /Applications/Codex.app
sh scripts/install_codex_app_patch_launch_agent.sh /Applications/Codex.app
```

当前已验证结论：
- 2026-04-25 已在真实 Codex.app 中验证：当前项目历史列表不再只显示 3 条
- 当前项目下 `custom / sub2api / openai` 的历史线程已可同时显示
- 当前 Codex Desktop / app-server 中，`thread/list` 的 `modelProviders: null` 不表示“全部 provider”；必须显式传 `modelProviders: []`

建议继续验收：
- 更换登录账号后历史列表仍可见
- 打开旧对话后可以继续发送消息
- Codex.app 更新后 LaunchAgent 能自动重打补丁

## Codex.app 打包补丁

当前 `/Applications/Codex.app` 是已打包 Electron 应用，不是源码仓库。为了让普通桌面端新线程和恢复线程默认启用更持久的本地历史，本项目提供补丁脚本：

```bash
node scripts/patch_codex_app_extended_history.js /Applications/Codex.app
```

只读检查：

```bash
node scripts/patch_codex_app_extended_history.js /Applications/Codex.app --check
```

脚本说明和执行记录见 `scripts/README.md`。

脚本行为：
- 备份 `Contents/Resources/app.asar`
- 备份 `Contents/Info.plist`
- 将普通新线程参数改为 `persistExtendedHistory: true`
- 将恢复线程参数改为 `persistExtendedHistory: true`
- 将最近线程列表参数改为 `useStateDbOnly: true` 且 `modelProviders: []`
- 将“全部线程”列表参数改为 `useStateDbOnly: true`
- 将项目入口最近线程列表参数改为 `useStateDbOnly: true` 且 `modelProviders: []`
- 更新 `Info.plist` 中的 `ElectronAsarIntegrity` hash
- 对修改后的 `.app` 做本地 ad-hoc 重新签名

关键兼容性结论：
- 当前 app-server 对 `thread/list` 的 `modelProviders` 语义与直觉不同
- `modelProviders: null` 只返回当前 provider 的线程
- `modelProviders: []` 才会返回全部 provider 的线程
- 因此补丁脚本必须把跨 provider 历史列表请求改成 `modelProviders: []`

桌面端自动更新会替换 `app.asar`，因此每次 Codex.app 更新后都需要重新执行该脚本。脚本是幂等的：若目标代码已经打过补丁，会保留现状并重新写入当前 hash。

若希望用户完全无感，可安装用户级 LaunchAgent：

```bash
sh scripts/install_codex_app_patch_launch_agent.sh /Applications/Codex.app
```

安装后会在登录时自动执行一次，并监听 `Contents/Resources/app.asar` 变化。Codex.app 更新覆盖补丁后，LaunchAgent 会自动重新运行补丁脚本。

## 不修改 App 的边界

不修改 Codex.app 时，本项目不会拦截 Codex.app 原生输入框，也不会强制替换 Codex.app 原生线程列表。因此它不满足“在 Codex.app 原生界面里直接看到全部历史”的核心需求。
